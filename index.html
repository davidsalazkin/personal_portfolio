Swim Log · A Python Web Application

Overview: I developed a web application using Python and Flask that allows users to keep track of their swimming workout history. Integrated SQLAlchemy toolkit to store registered accounts and swimming sessions. Incorporated bcrypt to hash passwords and Chart.js to neatly display data.

Utilized: Python HTML CSS JS SQL Flask Git

"Dashboard" page

"Add Swim" page

The Motive: Some studies show that swimming burns more calories than any other sport - football, soccer, and yes, the ever popular running. The benefits of swimming have been noted for some time and there are several mobile apps and web applications that allow users to log their workouts, but I wanted to tailor an application to my own needs. At the same time, I wanted to delve deeper into backend web development.

The Execution: This project was developed alongside a Python web development video series by Cory Schafer, but I wanted to retrofit my project with the ability for users to log swim data privately, view a set of analytics describing their data, and navigate an intuitive UI.

First, I wanted users to have the ability to register, login, and edit their personal information (email, password, and avatar). To do so I installed SQLAlchemy and developed a "User" model that would save their information to a SQLite database. I styled the login and registration forms with CSS powered by Bootstrap.

Once a user is able to login to their account, they want to be able to log their swim. I used a Flask extension, Flask-Login, to handle session management - this way I could hide and display necessary pages and information unique to users who are logged in.

Next, I developed another model for swimming sessions, along with a "Log Swim" form that allowed users to POST a "swim_session" to the database that then linked to them through a secondary key in "swim_session" and a primary key in "user". This is the meat of the application, and it took some time for me to be able to properly link the two models I had developed for my database since it was my first time working with SQL requests in a project.

With Flask-Login (session handling) and some table and form styling, I was now able to display to a user their logged information inside of a table (sorted by date). I also added a few forms and page templates that allow the user to edit or delete a session that they've logged. I wrapped this section up by adding pagination in case users had logged lots of swims.

Finally, I brought the whole project together by adding a Dashboard that includes a Javascript data-charting library called Chart.js. It allows the user to view their total swims, total time spent swimming, and a pie chart that displays the percentage of strokes they've swam.

Currently, the application only supports a total of up to five swims per session, but I would like to return to it and add a dynamic form option that let's users keep adding swims to a single session in the future.

LifeTime Password Gen · A Javascript Password Generator

Overview: I developed a password generator using Javascript that allows users to locally generate a secure password and then displays its phonetic spelling. Utilizes cookies to store the user's phonetic alphabet preferences.

Utilized: HTML CSS JS Git

"Password Generator" tab

"Phonetic Language Preference" tab

The Motive: I worked at a call center one summer with twelve others. We took approximately 1,000 calls a day from employees at Life Time, Inc. requesting to reset their passwords when the password reset web tool broke. I decided to put together a tool to help us in taking ~1,000 password reset calls every day. I wanted it to generate a secure password and print it's phonetic spelling, and I also wanted to get more familiar with Javascript in the process.

The Execution: The first thing I did was write a function that would generate the random password with the following criteria: eight characters long, one uppercase letter, one lowercase letter, and six numeric characters. I printed the phonetic spelling by pulling the terms from a dictionary object.

After going through a testing phase with my teammates in the call center, I decided to include an option for users to set their own phonetic language. Getting cookies to work was not as easy as I initially expected, I wanted to save all of the user's phonetic preferences at once. I ended up pulling the preferred terms from the relevant form upon submission on the password generation page, looping through each one to create a cookie, and then checking to see if there are saved cookies on page load. I set a default preset dictionary with the phonetic terms but I wrote a function that would overwrite it in the case that cookies were found.

Finally, we also received a relatively large amount of calls to reset Cisco phone PIN's as well, so I decided to re-engineer the password generator to add a PIN generator that would print six random numbers.
